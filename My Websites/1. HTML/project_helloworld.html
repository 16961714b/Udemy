Some interesting thoughts on debugging code

A software bug is an error, flaw, failure, or fault in a computer program or system that causes it to produce an incorrect or unexpected result, or to behave in unintended ways. (Source: Wikipedia)

The mindset and attitude to debugging is perhaps the most important part, because it determines how effectively you'll fix the error, and what you'll learn from it — if anything. (Source: StackExchange)

Classics on software development like The Pragmatic Programmer and Code Complete basically argue for the same approach: every error is an opportunity to learn, almost always about yourself (because only beginners blame the compiler/computer first). (Source: StackExchange)

So treat it as a mystery which will be interesting to crack. And cracking that mystery should be done systematically, by expressing our assumptions (to ourselves, or to others) and then testing our assumptions, one-by-one if need be — using every tool at our disposal, especially debuggers and automated test frameworks. Then after the mystery is solved, you can do even better by looking through all your code for similar errors you may have made; and write an automated test to ensure the error will not happen unknowingly again. (Source: StackExchange)

Bugs vs errors

Some would prefer the term "error" rather than "bug", because it points to programmer's own (sloppy) thinking, and not a "bug that maliciously sneaked in while the programmer was not looking" (E.W. Dijkstra).
Anyway, here is a list of terms that coders use while refering to bugs and errors:

  • problems
  • mistakes
  • anomalies
  • faults
  • failures
  • crashes
  • incidents
  • etc.

Three important things to share while asking for help

Learning how to learn is crucial for a successful coder. One of the biggest surprises for novice programmers is just how much of their "programming time" is spent as "debugging time." (Source: codeunion)

The need to establish a shared context

Student:

My code isn't working! What's wrong???
Teacher:
How am I supposed to know. I need more information. :-)
Can you please answer these 3 questions as best as you can, first?
  1. What makes you say your code isn't working?
  2. What did you expect your code to do and why?
  3. What did your code do instead and how do you know?

And an experienced programmer would usually answer along the lines:

"I wanted to do X, so I wrote <some-code>. I expected to see Y when I ran this code, but instead I saw Z. What am I missing?"
